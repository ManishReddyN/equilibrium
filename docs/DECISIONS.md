# Decisions Log

One line per deviation from `roommate-app-execution-plan.md`, with rationale. Newest entries at the bottom.

- **2026-07-18** — Execution environment is Windows 11 (no macOS/Xcode). iOS native builds, iOS Simulator runs, and the Fastlane iOS lane (`app/ios/fastlane/Fastfile`) cannot be executed on this machine under any configuration, including Docker: Apple's EULA prohibits virtualizing macOS on non-Apple hardware, and there is no Linux/Windows build of Xcode. Phase 0-4 iOS verification gates (`run-ios`, simulator checks) will be documented as blocked rather than run. iOS work is limited to writing source/config that a macOS machine could later build.
- **2026-07-18** — Ruby and JDK are not installed on this machine at plan start, so the Android toolchain and Fastlane (Ruby-based) cannot run yet either. Android-side verification gates are blocked until those are installed (a Docker image with JDK + Android SDK is a viable path for Android specifically, revisit at Phase 1/6).
- **2026-07-18** — Reusing the existing Supabase project from the prior `roomie` project (ref `zubxuqshcyniosmdztmw`) instead of provisioning a new Supabase project for Phase 2, per user direction. `roomie`'s existing schema (`households`, `roommates`, `chores`, `chore_log`) differs substantially from this plan's schema (`households`, `profiles`, `chores`, `assignments`, `audit_ledger`, `feedback_queue`, `market_listings`, `notification_outbox`). The exact merge/replace strategy for the live schema is deferred to Phase 2 and will be logged here when decided.
- **2026-07-18** — Executing phase-by-phase with a stop-and-report checkpoint after each Verification Gate, per user preference, rather than pushing through all phases autonomously.
- **2026-07-18** — Added `__tests__` to `app/tsconfig.json`'s `include` array (plan section 0.4 lists only `["src", "index.js", "*.config.js"]`). Without it, `@typescript-eslint/recommended-type-checked` (added in 0.5) cannot type-check `__tests__/App.test.tsx`, since Jest test files are program entry points that nothing else imports, so TS never pulls them into the type-checked program otherwise.
- **2026-07-18** — Replaced the RN CLI's default boilerplate `App.tsx` (which imports the untyped `react-native/Libraries/NewAppScreen` module) with a minimal typed placeholder, to satisfy strict typed-linting without suppression comments. It will be replaced again by the composed app shell in Phase 1/3 regardless.
- **2026-07-18** — Pinned all RN-template-generated `devDependencies` in `app/package.json` to their exact resolved lockfile versions (removed `^`/`~` ranges), per the plan's hard "pin every dependency" rule (section 0, item 4). The plan's Phase 1 install list only pins newly-added packages explicitly; this extends the same rule to the CLI-generated baseline.
- **2026-07-18** — Installed `babel-plugin-module-resolver@5.0.2` in Phase 0 (rather than its listed spot in Phase 1's batch install), since Phase 0 section 0.4 explicitly requires configuring it for path aliases immediately. Phase 1 will find it already present at the correct pinned version.
- **2026-07-18** — Prefixed the migration that drops `roomie`'s legacy schema as `0000_drop_legacy_roomie_schema.sql`, ahead of the plan's own `0001` starting point, so the destructive step is its own reviewable unit and sorts first without renumbering every subsequent migration the plan specifies.
- **2026-07-18** — `fn_next_handler` (rotation selection) is implemented as a bounded PL/pgSQL loop over `profiles.id`-ordered household members (advance from the current handler's index, skip a candidate only if a duo household member has a conflicting pending `is_debit_makeup` obligation for a different chore), rather than a single SQL expression, since the plan specifies the *behavior* (deterministic rotation + duo makeup-skip) but not a concrete algorithm. Verified empirically for both plain rotation and the duo skip/fallback case.
- **2026-07-18** — Notification fan-out (`fn_notify_new_assignment`, `fn_notify_new_listing`) is my own design: the plan requires `notification_outbox` and a `dispatch_mode` column but does not specify what populates it. Both triggers derive `dispatch_mode` from the household's `profile` (`co_living` → `digest`, otherwise `immediate`) and fire on `assignments`/`market_listings` insert; listing notifications fan out to every household member except the lister.
- **2026-07-18** — Added `fn_chores_baseline_immutable()` + `trg_chores_baseline_immutable` (migration 0004) to actually enforce the immutability that migration 0002's own comment on `chores.baseline_photo_path` ("immutable after onboarding (enforced by trigger)") promised but the plan never defined — raises if an existing non-null `baseline_photo_path` is changed.
- **2026-07-18** — Downgraded `prisma`/`@prisma/client` from `7.8.0` (initially installed as "latest") to the exact pin `6.19.3`. Prisma 7 removes `datasource.url` in favor of `prisma.config.ts` + driver adapters, which conflicts with the plan's literal `url = env("DATABASE_URL")` schema syntax and its `prisma db pull`/`generate` verification commands. 6.19.3 is the newest 6.x release and preserves the plan's exact commands.
- **2026-07-18** — Prisma's `multiSchema` (stable in 6.19.3) pulls in Supabase's *entire* `auth` schema, not just `auth.users`, because `profiles.id` has a cross-schema FK to `auth.users(id)` and Prisma's introspection has no per-table filtering. Accepted as an unavoidable tradeoff of vanilla Prisma introspection rather than hand-editing the generated schema (which `db pull` would just overwrite again).
- **2026-07-18** — The plan's literal verification command `prisma db pull --print | diff - schema.prisma` was replaced with `diff -B` (ignore blank-line-only changes): `--print` always emits 2 trailing blank lines that are never present in the saved file, causing a false-positive diff on an otherwise fully-synced schema.
- **2026-07-18** — Discovered that Supabase's default privilege bootstrap does **not** grant `authenticated`/`anon` any SELECT/INSERT/UPDATE/DELETE on tables created via SQL migrations (only REFERENCES/TRIGGER/TRUNCATE) — unlike tables created via the Studio UI. The plan's RLS policies (migration 0005) were syntactically correct but inert without a baseline privilege grant underneath them; every table now has explicit `grant` statements matching its policies. This was caught by the pgTAP suite (which hits tables as the literal `authenticated` role) rather than by earlier manual smoke tests, which only ever exercised `SECURITY DEFINER` RPCs and so never touched the missing grants.
- **2026-07-18** — Added `fn_handle_new_user()` + `trg_auth_user_created` (`after insert on auth.users`, migration 0004) to bootstrap a bare `profiles` row on signup. The plan specifies no `profiles` INSERT policy and no signup trigger anywhere (confirmed via grep across the full plan) — without this, no `profiles` row could ever exist for a new user, breaking the Phase 3 onboarding flow entirely.
- **2026-07-18** — Added `fn_create_household(p_name, p_roommate_count)` RPC (migration 0004), mirroring `fn_join_household`'s pattern, since the plan's function list only covers the join-existing-household onboarding path and not the create-new-household path that Phase 3's onboarding flow also requires.
- **2026-07-18** — Restricted `profiles`' UPDATE grant to a column allowlist (`full_name`, `avatar_path`, `push_token_ios`, `push_token_android`), deliberately excluding `household_id`/`cohort_index`/`points_balance`. The plan's literal `pr_update_self` policy (`using (id = auth.uid()) with check (id = auth.uid())`) has no column restriction, so as written it would let any authenticated user set their own `household_id` to any existing household, bypassing `fn_join_household`'s invite-code/capacity checks entirely. Closed via a column-level GRANT (same technique already used for `market_listings.lister_id` anonymity) rather than rewriting the plan's policy text, since narrowing write access is a lower-risk change than altering the specified policy predicate.
